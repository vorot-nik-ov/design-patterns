# Strategy (Стратегия)

Стратегия — это поведенческий паттерн проектирования, который определяет семейство алгоритмов, инкапсулирует каждый из них и делает их взаимозаменяемыми. Стратегия позволяет изменять алгоритмы независимо от объектов, которые их используют.

## Проблема
Вам нужно выполнять оплату разными способами (карта, наличные, криптовалюта, SberPay). Если написать всю логику в одном классе через `if/else` или `switch`, он превратится в огромный нечитаемый "God Object". При добавлении нового способа оплаты придется менять код основного класса, что нарушает Принцип открытости/закрытости (OCP).

## Решение
Вынести каждый способ оплаты в отдельный класс (стратегию), реализующий общий интерфейс `IPaymentStrategy`. Класс-контекст (`PaymentProcessor`) будет хранить указатель на текущую стратегию и делегировать ей выполнение оплаты. При необходимости способ оплаты можно легко заменить "на лету".

## Особенности реализации на C++
* **Делегирование:** Объект `PaymentProcessor` ничего не знает о деталях списания денег. Он просто вызывает метод `pay()` у текущей стратегии.
* **Смена стратегии в рантайме:** Метод `setStrategy()` позволяет переключаться между способами оплаты во время выполнения программы (как показано в `main()`).
* **Управление памятью:** В данном примере контекст берет на себя владение объектами стратегий (взяв указатель и удалив его в `setStrategy` и деструкторе). 
> *Примечание:* В современном C++ вместо сырых указателей и ручного `delete` здесь было бы элегантнее использовать `std::unique_ptr<IPaymentStrategy>`, но текущая реализация отлично демонстрирует сам суть паттерна и работу с памятью.

## Плюсы
* Изолирует код разных алгоритмов (способов оплаты) друг от друга.
* Замена алгоритмов во время выполнения программы без изменения контекста.
* Упрощает добавление новых стратегий (достаточно создать новый класс, реализующий интерфейс).

## Минусы
* Усложняет код за счет создания дополнительных классов.
* Клиент должен понимать разницу между стратегиями, чтобы выбрать правильную (нужно знать, когда вызывать `CardPayment`, а когда `CashPayment`).
* Если стратегий всего две и они почти не меняются, использование паттерна может быть избыточным.
