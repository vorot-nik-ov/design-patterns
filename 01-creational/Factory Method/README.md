# Factory Method (Фабричный метод)

Фабричный метод — это порождающий паттерн проектирования, который определяет общий интерфейс для создания объектов в суперклассе, но позволяет подклассам изменять тип создаваемых объектов.

## Проблема
Необходимо создавать различные типы объектов (например, факультеты университета), но клиентский код не должен зависеть от конкретных классов этих объектов. Если мы будем использовать оператор `new` прямо в клиентском коде, то при добавлении нового факультета придется менять код во множестве мест.

## Решение
Выделить общий интерфейс `IFaculty` и создать отдельный класс-фабрику `FacultyFactory`. Фабрика принимает перечисление (enum) с типом нужного объекта и возвращает готовый объект, скрытый за интерфейсом. Клиент работает только с интерфейсом и фабрикой.

## Особенности реализации на C++
* **Умные указатели:** Используется `std::unique_ptr` и `std::make_unique`. Это исключает ручное управление памятью (не нужно писать `delete`) и предотвращает утечки.
* **Безопасное перечисление:** Используется `enum class Type` вместо обычного `enum`. Это не дает случайно передать в фабрику обычное число (например, `5`) и заставляет явно указывать область видимости (`FacultyFactory::Type::Physics`).
* **Чистый интерфейс:** Класс `IFaculty` содержит чисто виртуальный деструктор и метод `showInfo()`, что делает его корректным абстрактным классом.

## Плюсы
* Изолирует клиентский код от конкретных классов продуктов (клиент не знает про `PhysicsFaculty`, он знает только про `IFaculty`).
* Упрощает добавление новых продуктов в программу (достаточно создать новый класс и добавить один `case` в фабрику).
* Обеспечивает автоматическое управление временем жизни объектов благодаря `unique_ptr`.

## Минусы
* При добавлении нового типа продукта все равно приходится модифицировать класс фабрики (добавлять новый `case` в `switch`), что нарушает принцип открытости/закрытости (OCP). Для строгого соблюдения OCP используют "Абстрактную фабрику" или наследуют сами фабрики.
