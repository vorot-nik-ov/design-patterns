# Proxy (Заместитель)

Заместитель — это структурный паттерн проектирования, который подставляет вместо реального объекта его специальный представитель (заместитель), перехватывающий вызовы и делегирующий их реальному объекту.

## Проблема
Обращение к тяжелому объекту (например, запрос к реальной базе данных `RealUserDatabase`) каждый раз, когда нужно получить данные, consumes много ресурсов и времени. Если данные запрашиваются часто и не меняются, это создает огромную нагрузку.

## Решение
Создать класс-заместитель (`UserDatabaseProxy`), который реализует тот же интерфейс, что и реальный объект. Заместитель хранит внутри себя указатель на реальную базу данных, а также локальный кэш. При запросе данных заместитель сначала проверяет кэш: если данные там есть и они не устарели, он возвращает их сразу. Если нет — делегирует запрос реальной базе, сохраняет результат в кэш и возвращает клиенту.

## Особенности реализации на C++
* **Единый интерфейс:** И реальная база данных (`RealUserDatabase`), и заместитель (`UserDatabaseProxy`) реализуют общий интерфейс `IUserDatabase`. Клиент не знает, с кем именно работает.
* **Кэширование с учетом времени:** В заместителе используется не только хранилище данных (`map<int, string>`), но и хранилище временных меток (`map<int, timestamp_>`) вместе с константой `CACHE_LIFETIME`. Это позволяет автоматически удалять устаревшие записи из кэша.
* **Ленивая инициализация (опционально):** Объект `RealUserDatabase` можно создавать не в конструкторе заместителя, а только в момент первого запроса, который не найден в кэше (что еще больше экономит ресурсы).

## Плюсы
* Существенно снижает нагрузку на сеть или базу данных благодаря кэшированию запросов.
* Позволяет контролировать доступ к реальному объекту (например, проверять права доступа перед выполнением запроса к БД).
* Работает даже если реальный объект еще не создан или недоступен.

## Минусы
* Усложняет код из-за введения дополнительных классов.
* Возможны проблемы с согласованностью данных (если данные в реальной базе изменились, а в кэше заместителя осталась старая версия до истечения `CACHE_LIFETIME`).
* Ответ замедляется из-за дополнительной проверки кэша (хотя это быстрее, чем поход в БД).
